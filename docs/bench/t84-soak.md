# T84: часовой soak/chaos-прогон, kill -9 под постоянной нагрузкой

**Проверяет:** задачу плана T84 "1 час прогон стресс-нагрузки (1000 ops/s, mixed
read/write) с периодическими kill -9 → отсутствие data loss, отсутствие
memory leak (RSS ровный), отсутствие Go-supervisor-zombie" и SPEC §7.5.

Архитектура: `SoakIT` (Java-клиент) гонит дозированную смешанную нагрузку (80% Put / 20%
Get, 4 writer-потока) по таблице `soak`, на которой висит counter-observer
coproc-jar с `hbasecop.policy.prePut=best-effort`; каждый rowkey подтверждённого клиентом Put
попадает в ledger, и финальный полный scan обязан содержать каждый
заведённый в ledger ключ. Вокруг этого `test/integration/scripts/soak.sh` крутит chaos-цикл
(`docker exec ... pkill -9 -f '[h]basecop-runtime'` с равномерно случайными
интервалами 120-300s, активируется только после появления runtime,
деактивируется за 30s до конца нагрузки) и sampler (каждые 10s: Go pid/RSS, RS JVM RSS, счётчик процессов
runtime, счётчик Z-состояний в CSV), после чего вычисляет gate'ы.

## Как воспроизвести

```
make soak                                    # full hour
make soak SOAK_DURATION_S=150 SOAK_KILL_MIN_S=25 SOAK_KILL_MAX_S=45 SOAK_RATE=400   # smoke
```

## Результат (2026-06-10)

Железо: AMD Ryzen 7 5800H, WSL2 + Docker, HBase 2.5.11 standalone.
Сконфигурированная нагрузка 3600s на 1000 ops/s, **впрыснуто 12 kill -9**.

| Gate | Результат | Подтверждение |
|------|--------|----------|
| Data loss | **PASS** | 2 839 779 подтверждённых Put, 2 839 779 прочитанных scan'ом строк, **0 потеряно**, 0 неудачных Put, 711 149 успешных Get |
| Go RSS ровный | **PASS** | медианы 13 718 → 13 946 kB (+1.7%, в полосе ≤15%) |
| RS JVM RSS ровный | **PASS** | медианы после ramp 888 672 → 946 372 kB (+6.5%, в полосе ≤10%); см. примечание |
| Один runtime | **PASS** | никогда не более 1 `hbasecop-runtime` в 345 сэмплах |
| Нет утёкшего runtime | **PASS** | 0 процессов runtime после drop таблицы |
| Zombie | **PASS** | 0 процессов в Z-состоянии на всём прогоне и в конце |
| Учёт restart'ов | **PASS** | `GoProcess started:` ×13 = 1 начальный + 12 kill, ровно столько; 0 `UNHEALTHY` |

Примечательно: **ноль видимых клиенту отказов** во всех 12 окнах падений.
best-effort-политика + 3s restart-deadline поглотили каждый kill без единого
отклонённого Put, и каждый restart был 1:1 (никаких restart-штормов, ни одного
пропущенного падения).

## Выбор окна для RS JVM RSS

JVM RegionServer'а раскачивается с 714 → 871 MB за первые ~25 минут (расширение heap,
заполнение block-cache и memstore по мере приёма 2.8M строк), а
затем выходит на плато, поэтому сравнение первого окна с последним принимает прогрев за
утечку. Поэтому gate сравнивает окно 60-80% с финальными 20%
(измеренный дрейф: 6.5%); реальная утечка Java-bridge под постоянной нагрузкой держит
свой наклон и проваливает это окно. Go-процесс (который bridge перезапускает
на каждый kill) гейтится по схеме первое-против-последнего на ±15% и оставался абсолютно ровным.

## Баги харнесса, пойманные smoke-прогонами (исправлены до часового прогона)

- `docker exec sh -c 'pgrep -f hbasecop-runtime'` матчит cmdline самой
  зондирующей shell, поэтому каждый gate подсчёта процессов проваливался детерминированно.
  Исправлено bracket-паттерном `[h]basecop-runtime`.
- kill'ы до появления Go runtime (кластер ещё загружается) и kill'ы,
  попадающие в пустые restart-окна, учитывались, ломая учёт
  restart'ов. Теперь chaos активируется только после появления runtime и считает
  только те kill'ы, что нашли живую жертву.

## Оговорки

- Топология standalone с одним RS; утверждения "per RegionServer" проверяются на
  ровно одном RS. Chaos с несколькими RS — задача на будущее.
- Каденция kill'ов (120-300s) ограничивает обнаружимую скорость утечек на Go-стороне: процесс
  никогда не живёт дольше ~5 мин, поэтому более медленные Go-утечки прячутся за restart'ами
  (у оси RS-JVM такого слепого пятна нет; её никогда не перезапускают).
- Heartbeat'ы отключены в конфиге counter-observer; детекция падений
  шла по пути crash-probe в 500ms (фикс C2). Путь детекции зависаний
  проверяется юнит-тестами, а не этим soak.
