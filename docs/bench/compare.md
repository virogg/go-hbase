# Native vs Go: сравнение копроцессоров (эквивалентность + perf)

**Проверяет:** что копроцессор с одной и той же логикой, написанный (а) нативно
на Java и (б) на Go поверх bridge'а, даёт **идентичные результаты**, и измеряет
**накладные расходы** Go-плеча относительно нативного — на четырёх типах
копроцессоров и действий.

В отличие от T81/T82 (которые меряют *чистую стоимость bridge'а* против no-op
baseline'а), здесь оба плеча выполняют **реальную, эквивалентную логику**:
нативный Java-копроцессор — это не no-op, а полноценная реализация той же
функции. Так что дельта — это «та же логика в Go-через-shmem против нативного
JVM», а равенство результатов — независимая проверка корректности bridge'а
end-to-end.

## Что сравнивается

Каждая пара — нативный класс из `examples/native-coproc` против Go-примера, оба
зарегистрированы per-table на двух co-resident таблицах (`*_native` / `*_go`) в
одном кластере; один и тот же клиентский путь, один и тот же workload.

| Сравнение | Тип / хук | Логика | Go-плечо | Нативное плечо |
|---|---|---|---|---|
| **sum** | endpoint (read, агрегация) | scan региона, сумма `cf:n` как int64 | `endpoint-observer` (`GenericRegionObserver`) | `NativeSumEndpoint` |
| **ttl** | observer `prePut` (write, валидация) | reject Put без `ttl=<sec>;` конверта | `ttl-validator` (`TtlValidatorRegionObserver`) | `NativeTtlObserver` |
| **audit** | observer `postPut` (write, CPU) | SHA-256-дайджест строки + счёт ячеек в лог | `audit-observer` (`AuditRegionObserver`) | `NativeAuditObserver` |
| **filter** | observer `preGetOp` (read, bypass) | bypass Get'а по префиксу строки `block-` | `filter-observer` (`FilterRegionObserver`) | `NativeFilterObserver` |

Нативный endpoint переиспользует **тот же** `GoEndpointService`-surface, что и
Go-плечо, поэтому клиентский путь (`EndpointClient.callAndReduce`) байт-в-байт
одинаков для обоих — различается лишь серверная реализация Service'а
(нативный Java против форвардинга в Go через shmem).

## Как воспроизвести

```
make compare-all                     # все четыре, один boot кластера
# по отдельности:
make compare-sum
make compare-ttl
make compare-audit
make compare-filter
```

Каждый таргет стейджит `native-coproc.jar` плюс нужный Go-пример в
`test/integration/coproc-jars/`, поднимает dockerized HBase 2.5 standalone,
гоняет соответствующий `*CompareIT`, сбрасывает лог контейнера в
`test/integration/coproc-jars/hbase-compare-*.log` и гасит кластер.

## Методология

- **Эквивалентность (жёсткий pass/fail).** Оба плеча прогоняются на идентичных
  данных, результат сверяется `assertEquals`:
  - *sum*: `nativeSum == goSum == ` арифметическая сумма засеянной `cf:n`;
  - *ttl*: для корпуса валидных/невалидных значений оба плеча принимают
    решение accept/reject **идентично** (и состояние таблицы совпадает);
  - *audit*: оба плеча эмитят одинаковый `row_digest` для каждой строки (и он
    совпадает с независимым Java-оракулом SHA-256); сверка через скрейп
    `docker logs`;
  - *filter*: blocked-Get пуст, allowed-Get присутствует — одинаково на обоих.
- **Производительность (только отчёт, без гейта).** Чередующиеся A/B-повторы
  (native-first / go-first попеременно, чтобы погасить дрейф), медиана по
  раундам. Печатаются машиночитаемые строки `COMPARE_RESULT` (per-arm) и
  `COMPARE_SUMMARY` (native/go медианы + отношение `go_over_native`). Числа
  шумны на shared-runner'ах CI и **не гейтят** сборку — это измерение, как
  bench-jobs.

## Результат (`make compare-all`, один boot, dev-хост, HBase 2.5.11)

| Сравнение | native median | go median | `go_over_native` | Δ (go−native) | equivalent |
|---|---|---|---|---|---|
| sum (агрегация, 2000 строк) | 4.9 ms | 7.7 ms | **1.56×** | +2.8 ms / вызов | ✅ |
| filter (1 blocked + 1 allowed Get) | 2.4 ms | 3.8 ms | **1.59×** | +1.4 ms / 2 Get | ✅ |
| audit (батч 500 postPut) | 11.4 ms | 65.2 ms | **5.70×** | +108 µs / put | ✅ |
| ttl (батч 500 prePut) | 7.7 ms | 63.8 ms | **8.28×** | +112 µs / put | ✅ |

### Как это читать

- **Корректность совпадает во всех четырёх случаях** — bridge доставляет ту же
  семантику, что и нативный копроцессор: ту же агрегацию, те же accept/reject
  решения, те же дайджесты, тот же read-bypass.
- **Read-path / endpoint (sum, filter): Go в пределах ~1.5–1.6× нативного.** Одна
  агрегация или один Get амортизируют один bridge round-trip по многим
  ячейкам/строкам, так что относительная стоимость shmem-перехода невелика.
- **Write-path per-op хуки (ttl, audit): Go ~5–8× нативного.** Каждый Put
  синхронно пересекает shmem-кольцо в `prePut`/`postPut`: в батче из 500 записей
  доминирует per-Put round-trip — там, где нативный хук это вызов метода в том же
  JVM. **Устойчивая** величина здесь — не отношение, а **абсолютная дельта
  ~110 µs/put** (один синхронный межпроцессный round-trip): она одинакова для ttl
  и audit. Отношение же шумит (ttl наблюдалось от ~5× до ~8×), потому что
  знаменатель — нативное время записи (memstore/WAL) — само пляшет 7–12 ms.
- **Почему read дёшев, а per-op write дорог — одно и то же объяснение:** стоимость
  процессной границы платится за *пересечение*, а не за *операцию*. Read/endpoint
  пересекает границу один раз и амортизирует; per-op write пересекает её на каждый
  Put. Снизить write-overhead к ~1.x можно, перенеся валидацию на батч-хук
  (`preBatchMutate`, одно пересечение на батч — на обоих плечах, чтобы остаться
  apples-to-apples) и/или сделав best-effort post-хуки асинхронными (round-trip
  уходит с критического пути). Это меняет *как часто* платится граница, а не
  пытается сделать IPC бесплатным.

Числа — один прогон на одной машине; воспроизводятся теми же `make`-таргетами.
Абсолютные значения зависят от хоста; устойчивы порядок read-овэрхеда (~1.5×) и
абсолютная per-op дельта write-пути (~110 µs).
